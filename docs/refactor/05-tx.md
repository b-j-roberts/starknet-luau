## 5. tx/

### CallData.luau
_158 lines. Encode Luau values into Starknet calldata (flat felt arrays). All values are hex strings with "0x" prefix._

- `[ok]` Clean, focused module. Each encoder has a single responsibility and returns `{ string }` consistently.
- `[ok]` `encodeMulticall()` correctly implements Cairo 1 `__execute__` calldata format with length-prefixed per-call calldata.
- `[ok]` `encodeU256()` correctly splits into `[low, high]` using bit masking, matching starknet.js behavior.
- `[ok]` `encodeShortString()` validates both length (≤31) and ASCII range (≤127), throwing structured errors.
- `[ok]` `MASK_128` pre-computed once at module load — correct pattern for constant BigInt values.

- `[refactor]` **`encodeShortString()` duplicated in `wallet/TypedData.luau:45-54`.** TypedData has its own `local function encodeShortString(str)` with identical logic minus the validation guards (no >31 or non-ASCII checks). This means TypedData silently accepts invalid short strings. CallData.encodeShortString should be the single source of truth — TypedData should import it. The return type differs slightly (CallData returns `{ string }`, TypedData returns bare `string`), so CallData could expose a bare-string variant or TypedData could index `[1]`.

- `[refactor]` **`normalizeHex()` duplicated across 3 modules.** Three independent implementations exist:
  1. `CallData.luau:36-38` — `BigInt.toHex(BigInt.fromHex(hex))` (roundtrip through BigInt)
  2. `OutsideExecution.luau:134-136` — identical to CallData (BigInt roundtrip)
  3. `PaymasterPolicy.luau:36-45` — manual string manipulation (lowercase, strip "0x", strip leading zeros, re-add "0x")

  These should be consolidated into a single `normalizeHex()` in a shared utility (or added to `BigInt.normalizeHex()`). The PaymasterPolicy version is also semantically different — it doesn't round-trip through BigInt, so edge cases may diverge.

- `[refactor]` **`encodeStruct()` is a no-op wrapper.** Lines 101-107: copies input array into a new array element-by-element. Since the input is already `{ string }`, this is equivalent to `table.clone(fields)`. The function exists for API symmetry with `encodeArray()`, but the copy is wasteful. Consider either (a) returning the input directly (callers shouldn't mutate calldata) or (b) using `table.clone()` for clarity.

- `[api]` **`numberToHex()` uses StarkField for a non-field operation.** Line 141-143: converts a number to hex via `StarkField.fromNumber()` which reduces mod P. For numbers that fit in a felt this is fine, but the function name suggests general purpose. Should either be renamed to `feltFromNumber()` or use `BigInt.toHex(BigInt.fromNumber(n))` to avoid the implicit field reduction.

- `[type]` **`Call` type has no validation for `contractAddress` format.** The type accepts any `string` — callers can pass non-hex, non-0x-prefixed, or empty strings. Validation only happens downstream in `encodeMulticall()` where `normalizeHex()` will throw cryptic BigInt errors. Consider adding an explicit `validateCall()` helper or documenting the expected format in the type comment.

- `[test]` 61 tests — comprehensive coverage of all public functions. Cross-references starknet.js expected values.
- `[test]` **No negative tests for `encodeMulticall()` with malformed Call objects.** Missing: empty contractAddress, empty entrypoint, nil calldata field.

### TransactionHash.luau
_250 lines. V3 INVOKE and DEPLOY_ACCOUNT transaction hash computation using Poseidon (SNIP-8)._

- `[ok]` Clean separation: internal helpers operate on `Felt` (buffer), public API accepts/returns hex strings.
- `[ok]` `encodeResourceBoundInternal()` correctly packs `(name << 192) | (amount << 128) | price` per SNIP-8 spec.
- `[ok]` `hexArrayToFelts()` is a clean helper bridging hex-string API boundary to Felt-based Poseidon hashing.
- `[ok]` Both `calculateInvokeTransactionHash` and `calculateDeployAccountTransactionHash` have comprehensive default handling for optional fields.
- `[ok]` Constants (`INVOKE_PREFIX`, `DEPLOY_ACCOUNT_PREFIX`, `V3_VERSION`, gas names) are pre-computed as Felts at module load — correct for hot-path performance.
- `[ok]` Chain ID constants (`SN_MAIN`, `SN_SEPOLIA`) are correctly exposed as hex strings for consumer use.

- `[refactor]` **CRITICAL — Fee field hash computation duplicated inline in both `calculateInvokeTransactionHash` (lines 131-148) and `calculateDeployAccountTransactionHash` (lines 202-219).** Identical 18 lines: parse tip, encode 3 resource bounds, call `Poseidon.hashMany`. This is exactly what `hashFeeField()` (lines 90-102) already does — but the two calculate* functions don't call it. They inline the same logic. The public `hashFeeField()` should be called internally to eliminate this duplication:
  ```luau
  -- In both calculate* functions, replace 18 lines with:
  local feeFieldHash = StarkField.fromHex(TransactionHash.hashFeeField(tip, params.resourceBounds))
  ```

- `[refactor]` **DA mode encoding duplicated.** Both calculate* functions call `encodeDAModeInternal()` identically (lines 158 and 228). Not a large duplication, but combined with the fee field duplication above, the two functions share ~25 lines of identical preamble code. Consider extracting a `prepareCommonFields(params)` helper that returns `{ feeFieldHash, paymasterHash, daMode }`.

- `[refactor]` **Default parameter extraction duplicated.** Lines 124-129 and 196-200 are identical: `local versionFelt = ...; local tip = params.tip or "0x0"; ...`. Another ~6 lines of boilerplate shared between the two functions.

- `[type]` **`ResourceBounds` has 3 fields (`l1Gas`, `l2Gas`, `l1DataGas`) but `RpcTypes.ResourceBounds` has only 2 (`l1_gas`, `l2_gas`).** The `l1DataGas` / `l1_data_gas` field is missing from `RpcTypes.ResourceBounds` (line 207-210 of RpcTypes.luau). This means:
  1. The RPC wire type doesn't include L1 data gas in the resource bounds sent to the node
  2. `toRpcResourceBounds()` in TransactionBuilder.luau (lines 132-143) drops `l1DataGas` entirely — it only maps `l1Gas` and `l2Gas`
  3. The hash computation **does** include `l1DataGas` (correctly, per SNIP-8)
  4. This is likely correct per current Starknet RPC spec (L1 data gas is not user-specified in the RPC), but the inconsistency between hash types and wire types should be **documented explicitly** and `RpcTypes.ResourceBounds` should either include the field (optional) or have a comment explaining the omission.

- `[api]` **`encodeResourceBound()` (line 84) accepts `resourceName: string` (raw hex) while `hashFeeField()` (line 90) handles names internally.** These two public functions have inconsistent abstraction levels. `encodeResourceBound` requires the caller to know the hex-encoded ASCII name; `hashFeeField` hides this detail. For SDK consumers, `encodeResourceBound` is too low-level — they should rarely need to call it directly. Consider making it internal-only (unprefixed local function).

- `[test]` 80+ tests — excellent coverage of both hash functions with cross-reference vectors from starknet.js.
- `[test]` **No fuzz/property test for hash determinism.** Current tests use fixed vectors. A property test verifying `hash(same_inputs) == hash(same_inputs)` across random inputs would catch accidental state leakage.

### TransactionBuilder.luau
_595 lines. High-level transaction orchestration: nonce → fee estimation → hash → sign → submit._

- `[ok]` Clean orchestration pattern: each method follows a linear step sequence with clear comments.
- `[ok]` `applyMultiplier()` correctly handles zero values and uses integer arithmetic (scale=100) to avoid floating-point imprecision.
- `[ok]` `resourceBoundsFromEstimate()` correctly maps RPC fee estimate fields to internal ResourceBounds format.
- `[ok]` `buildInvokeTransaction()` and `buildDeployAccountTransaction()` produce correctly-shaped RPC objects.
- `[ok]` `dryRun` mode correctly executes the full pipeline (nonce, fee, hash, sign) but skips submission — good for testing and debugging.
- `[ok]` Fee estimation with `SKIP_VALIDATE` flag and dummy signature `["0x0", "0x0"]` follows starknet.js convention.
- `[ok]` `maxFee` cap logic (lines 504-521) correctly divides by price to find max amount, then clamps — arithmetic is sound.
- `[ok]` Promise acquisition via `provider:_getPromise()` enables both Roblox and Lune environments.

- `[fix]` **DA modes hardcoded to `"0x0"` in `buildInvokeTransaction` (lines 168-169) and `buildDeployAccountTransaction` (lines 199-200), ignoring the `nonceDataAvailabilityMode`/`feeDataAvailabilityMode` params.** The function signatures accept these parameters (lines 155-156, 184-185) and the `execute()`/`deployAccount()` methods pass them (via opts), but the builder functions ignore them and hardcode `"0x0"`. This means:
  1. Setting `nonceDataAvailabilityMode = 1` or `feeDataAvailabilityMode = 1` in `ExecuteOptions` has **no effect** on the submitted transaction
  2. The **hash computation** correctly uses the DA mode values (via `TransactionHash`), so the hash and the submitted transaction would mismatch if non-zero DA modes were used
  3. This is a **correctness bug** that would cause transaction rejection on any chain that supports non-L1 DA modes

  Fix:
  ```luau
  -- In buildInvokeTransaction:
  nonce_data_availability_mode = string.format("0x%x", params.nonceDataAvailabilityMode or 0),
  fee_data_availability_mode = string.format("0x%x", params.feeDataAvailabilityMode or 0),

  -- Same for buildDeployAccountTransaction
  ```

- `[refactor]` **CRITICAL — `execute()` and `deployAccount()` share ~80% identical flow.** Both methods follow the same pattern:
  1. Resolve nonce (fetch or override)
  2. Fetch chain ID
  3. Determine resource bounds (estimate or override, with multiplier)
  4. Compute transaction hash
  5. Sign (`BigInt.fromHex(txHash)` → `signer:signTransaction()`)
  6. Build signed transaction
  7. Handle dryRun
  8. Submit

  The differences are:
  - `execute` fetches nonce from provider; `deployAccount` defaults to `"0x0"`
  - Different hash functions and builder functions
  - `deployAccount` has `maxFee` cap logic
  - Different submission methods (`addInvokeTransaction` vs `addDeployAccountTransaction`)

  This should be refactored into a shared internal pipeline:
  ```luau
  local function executePipeline(provider, account, config: {
      resolveNonce: () -> string,
      encodeCalldata: () -> { string },
      computeHash: (chainId, nonce, resourceBounds, ...) -> string,
      buildTransaction: (params) -> any,
      submitTransaction: (signedTx) -> any,
      postEstimate: ((resourceBounds) -> ResourceBounds)?,
  }, options): any
  ```

- `[refactor]` **`estimateFee()` and `estimateDeployAccountFee()` also share ~80% identical logic.** Both:
  1. Build unsigned tx with zero resource bounds + dummy signature
  2. Call `provider:estimateFee({ unsignedTx }, { "SKIP_VALIDATE" })`
  3. Extract first estimate from result

  The only differences are the builder function and the nonce source. Could be extracted to:
  ```luau
  local function estimateInternal(provider, buildUnsignedTx: () -> any): any
  ```

- `[refactor]` **Fee estimate result extraction duplicated 4 times.** Lines 266-270, 339, 437-441, and 500 all do:
  ```luau
  if type(feeResult) == "table" and #feeResult > 0 then X = feeResult[1] else X = feeResult end
  ```
  Extract to: `local function extractFirstEstimate(feeResult)`.

- `[refactor]` **`ZERO_RESOURCE_BOUND` constructed 4 times inline.** Lines 248-252, 320-325, 419-423, and 482-487 all build the same `{ l1Gas = ZERO_RESOURCE_BOUND, l2Gas = ZERO_RESOURCE_BOUND, l1DataGas = ZERO_RESOURCE_BOUND }` table. Define a module-level `ZERO_RESOURCE_BOUNDS` (plural) constant:
  ```luau
  local ZERO_RESOURCE_BOUNDS: TransactionHash.ResourceBounds = {
      l1Gas = ZERO_RESOURCE_BOUND,
      l2Gas = ZERO_RESOURCE_BOUND,
      l1DataGas = ZERO_RESOURCE_BOUND,
  }
  ```

- `[refactor]` **Promise + pcall boilerplate repeated in every method.** Every async method wraps its entire body in:
  ```luau
  return Promise.new(function(resolve, reject)
      local ok, result = pcall(function() return provider:xxx():expect() end)
      if not ok then reject(result) return end
      ...
  end)
  ```
  Consider a helper:
  ```luau
  local function promiseStep(fn): (ok: boolean, result: any)
      return pcall(function() return fn():expect() end)
  end
  ```
  Or better: chain Promises natively instead of using `pcall + :expect()` inside `Promise.new()`. The current pattern converts async Promises to sync (`:expect()`) and back — it works, but it's an anti-pattern that masks the true async flow.

- `[refactor]` **`toRpcResourceBounds()` drops `l1DataGas` silently.** Lines 132-143 only map `l1Gas` and `l2Gas` to the RPC format, omitting `l1DataGas`. This is probably correct per current RPC spec (see TransactionHash note above), but a comment explaining the intentional omission is needed. If a future RPC version adds `l1_data_gas`, this will be a silent data loss bug.

- `[refactor]` **`maxFee` cap logic only exists in `deployAccount()`, not in `execute()`.** Lines 504-521. If a consumer wants to cap fees on a regular invoke, they must manually compute resource bounds themselves. This is an inconsistency — either both methods should support `maxFee` or neither should (with the consumer expected to pre-compute bounds). `ExecuteOptions` already has `maxFee: string?` in the type (line 27), but `execute()` never reads it.

- `[type]` **`Account` type (lines 16-21) is a minimal interface, but it's defined here AND consumed in `wallet/Account.luau`.** The `wallet/Account` class structurally conforms to this interface but doesn't import or reference it. This creates a fragile duck-typing contract. If `TransactionBuilder.Account` changes (e.g., adds a field), `wallet/Account` may silently break. Consider having `wallet/Account` explicitly declare conformance or importing the type.

- `[type]` **All 5 public methods return `any`.** `estimateFee`, `execute`, `estimateDeployAccountFee`, `deployAccount`, `waitForReceipt` all return `any` instead of typed Promises. This is a known Luau limitation with metatable classes, but it means consumers get zero autocomplete on result types.

- `[type]` **`ExecuteOptions` and `DeployAccountOptions` share 8 identical fields.** Lines 25-37 and 46-58. Only differences: DeployAccountOptions has `waitForConfirmation` and lacks `accountDeploymentData`. Extract a shared base type:
  ```luau
  type BaseTransactionOptions = {
      nonce: string?,
      maxFee: string?,
      resourceBounds: TransactionHash.ResourceBounds?,
      feeMultiplier: number?,
      tip: string?,
      paymasterData: { string }?,
      nonceDataAvailabilityMode: number?,
      feeDataAvailabilityMode: number?,
      dryRun: boolean?,
      skipValidate: boolean?,
  }
  ```
  Then `ExecuteOptions` and `DeployAccountOptions` can extend it (Luau intersection types).

- `[api]` **`skipValidate` defaults to `true` in `execute()` (line 329) but the type says `boolean?`.** The comment says "Use SKIP_VALIDATE flag for fee estimation" but the default behavior is to always skip validation. This should be documented more explicitly — consumers may not realize validation is being skipped by default.

- `[api]` **`waitForReceipt` (line 590) is a thin pass-through** to `provider:waitForTransaction()`. It adds no logic and exists solely for API symmetry. This is fine as a convenience method, but it should be documented as a delegation. The Account class also wraps this (Account:waitForReceipt), creating a 3-layer delegation chain: `Account → TransactionBuilder → Provider`.

- `[test]` 50+ tests for execute, plus 100+ tests in DeployAccount.spec.luau — comprehensive.
- `[test]` **No test for the DA mode bug.** No test passes `nonceDataAvailabilityMode = 1` and verifies the submitted transaction includes `nonce_data_availability_mode = "0x1"`. This would catch the hardcoded `"0x0"` bug.
- `[test]` **No test for `maxFee` cap in `execute()`.** Since `execute()` ignores `maxFee` (unlike `deployAccount()`), there should be a test documenting this gap or the feature should be implemented.

### Priority Actions (tx/)

1. **HIGH — Fix DA mode bug in `buildInvokeTransaction` and `buildDeployAccountTransaction`.** Lines 168-169 and 199-200 hardcode `"0x0"` instead of using the params. This causes hash/transaction mismatch for any non-L1 DA mode usage. Fix is a 4-line change. Add corresponding test.

2. **HIGH — Call `hashFeeField()` internally in both `calculate*TransactionHash` functions.** The public function exists and is tested, but the two hash functions inline the same logic. This is a correctness risk — if the fee field encoding changes, the inline copies could diverge from the canonical `hashFeeField()`.

3. **HIGH — Implement or remove `maxFee` support in `execute()`.** `ExecuteOptions.maxFee` exists in the type definition (line 27) but is never read in `execute()`. Either implement the cap logic (copy from `deployAccount`) or remove the field from the type to avoid misleading consumers.

4. **MEDIUM — Extract shared transaction pipeline.** `execute()` and `deployAccount()` share ~80% of their code. Extract a `_executePipeline()` internal helper parameterized by the transaction-specific parts (hash function, builder, submitter, nonce strategy). This eliminates ~100 lines of duplication and makes adding future transaction types (e.g., DECLARE) trivial.

5. **MEDIUM — Extract shared estimate pipeline.** `estimateFee()` and `estimateDeployAccountFee()` share ~80% of their code. Extract `_estimateInternal()` parameterized by builder function and nonce source.

6. **MEDIUM — Consolidate `encodeShortString()` and `normalizeHex()` across modules.** Move `normalizeHex` to a shared utility (e.g., `src/utils/hex.luau` or add `BigInt.normalizeHex()`). Have TypedData import `CallData.encodeShortString` instead of duplicating it. This eliminates 3 separate `normalizeHex` implementations and 1 `encodeShortString` duplicate.

7. **MEDIUM — Define `ZERO_RESOURCE_BOUNDS` (plural) module constant.** Replace 4 inline constructions with a single constant reference.

8. **MEDIUM — Extract `extractFirstEstimate()` helper.** Replace 4 identical conditional extractions with a single function call.

9. **MEDIUM — Extract `BaseTransactionOptions` type.** Deduplicate 8 shared fields between `ExecuteOptions` and `DeployAccountOptions`.

10. **MEDIUM — Document `l1DataGas` omission in `toRpcResourceBounds()` and `RpcTypes.ResourceBounds`.** Add comments explaining that L1 data gas is included in hash computation but not in the RPC wire format, per current Starknet spec.

11. **LOW — Consider making `encodeResourceBound()` internal.** It's a low-level encoding detail that SDK consumers shouldn't need. If kept public, document when a consumer would use it.

12. **LOW — Fix `numberToHex()` to use BigInt instead of StarkField.** Avoids implicit field reduction for a utility function with a general-purpose name.

13. **LOW — Update barrel comment from "V3 INVOKE" to "V3 INVOKE and DEPLOY_ACCOUNT".**
